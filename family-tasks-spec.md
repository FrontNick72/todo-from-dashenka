# Спецификация iOS-приложения «Семейные задачи»

> Документ предназначен для передачи в **Claude Code** как техническое задание.
> Проза — на русском, все идентификаторы в коде (поля, коллекции, функции) — на английском, как указано ниже.

---

## 1. Цель и краткое описание

iOS-приложение для двух пользователей (супруги), в котором они **симметрично** ставят
друг другу задачи. Задача может быть двух типов: привязана к конкретной дате/времени
либо иметь дедлайн, отсчитываемый от момента создания. Второй пользователь получает
**мгновенное уведомление** о новой задаче, изменении статуса, переносе даты и новых
комментариях — даже когда приложение закрыто.

Задачам можно назначать **приоритет** и **теги/категории**, а к задачам и комментариям
прикреплять **фотографии**.

Ключевые сценарии:
- Пользователь A ставит задачу пользователю B → B получает push.
- Исполнитель отмечает статус (выполнено / не выполнено) и оставляет комментарии → автор получает push.
- Исполнитель может перенести дату, но **только с обязательным комментарием-причиной** → автор видит новую дату и причину, получает push.
- Оба видят полную историю изменений по каждой задаче.

## 2. Технологический стек

| Слой | Технология |
|---|---|
| Язык / UI | Swift + SwiftUI (iOS 16+) |
| Локальное хранилище/кэш | Firestore offline persistence (встроено) |
| Облако (данные) | Cloud Firestore |
| Хранилище фото | Cloud Storage for Firebase |
| Аутентификация | Firebase Auth (email/пароль на MVP; Sign in with Apple — опционально позже) |
| Push-уведомления | Firebase Cloud Messaging (FCM) + APNs |
| Серверная логика для push | Cloud Functions for Firebase (Node.js/TypeScript) |
| Локальные напоминания | `UserNotifications` (UNUserNotificationCenter) |
| Выбор / съёмка фото | PhotosUI (PhotosPicker) + камера |
| Управление зависимостями | Swift Package Manager |

---

## 3. Архитектура

```
[iPhone A]  <—— realtime listener ——>  [ Cloud Firestore ]  <—— realtime listener ——>  [iPhone B]
     |                                        |
     | фото ⇅                                 | триггер onWrite
     v                                        v
 [ Cloud Storage ]                    [ Cloud Functions ]
                                              |
                                              v
                                     [ FCM ] ——> [ APNs ] ——> push на телефон партнёра
```

Два механизма доставки изменений — это важно понимать:

1. **Realtime-листенеры Firestore** — когда приложение открыто, второе устройство
   видит изменения мгновенно, без всяких push. Работает на бесплатном тарифе.
2. **Push через Cloud Functions + FCM** — нужны, чтобы уведомление пришло, когда
   приложение **закрыто или в фоне**. Именно ради этого нужен серверный компонент.

Фотографии хранятся в **Cloud Storage**; в Firestore лежат только метаданные и ссылки.

---

## 4. Важное про стоимость (прочитать до начала)

- **Cloud Functions требуют тарифа Firebase Blaze** (pay-as-you-go, нужна привязка
  банковской карты). Бесплатный тариф Spark их деплоить не даёт.
- Тариф Blaze **включает бесплатные лимиты**, и для двух пользователей реальное
  потребление в них укладывается — по факту счёт будет **$0/мес**. Карта нужна только
  как гарантия на случай превышения (которого не будет).
- **Cloud Storage** входит в тот же проект Blaze. Объём фото у двух человек ничтожен
  относительно бесплатных лимитов, поэтому тоже фактически бесплатно. Обязательно
  сжимать фото перед загрузкой (см. раздел 6), чтобы не раздувать трафик и хранилище.
- Единственная гарантированная плата — **Apple Developer Program, $99/год**
  (без него не работают push на реальном устройстве и распространение через TestFlight).

## 5. Модель данных (Firestore)

Структура коллекций (subcollections для комментариев, истории и вложений):

```
users/{uid}
  - displayName: string
  - email: string
  - fcmTokens: string[]        // токены устройств (могут ротироваться, хранить массивом)
  - spaceId: string | null     // общее пространство пары
  - createdAt: timestamp

spaces/{spaceId}
  - memberUids: string[]       // ровно два uid
  - inviteCode: string         // код для привязки второго участника
  - createdAt: timestamp

spaces/{spaceId}/tags/{tagId}          // общий справочник тегов/категорий пары
  - name: string
  - color: string              // hex, напр. "#FF9500"
  - createdBy: string
  - createdAt: timestamp

spaces/{spaceId}/tasks/{taskId}
  - title: string
  - notes: string | null
  - createdBy: string          // uid автора
  - assignedTo: string         // uid исполнителя
  - type: "scheduled" | "deadline"
  - startAt: timestamp         // для deadline = момент создания; для scheduled = момент создания
  - dueAt: timestamp           // дата/время или дедлайн
  - status: "open" | "completed" | "declined"
  - priority: "low" | "normal" | "high"   // по умолчанию "normal"; UI-название поля — «Степень важной важности»
  - tagIds: string[]           // ссылки на spaces/{spaceId}/tags
  - hasAttachments: boolean    // денормализованный флаг для списка
  - createdAt: timestamp
  - updatedAt: timestamp

spaces/{spaceId}/tasks/{taskId}/comments/{commentId}
  - authorUid: string
  - text: string
  - createdAt: timestamp

spaces/{spaceId}/tasks/{taskId}/events/{eventId}   // история изменений
  - type: "created" | "status_changed" | "date_changed" | "commented" | "attachment_added"
  - byUid: string
  - oldValue: string | null    // напр. старая дата в ISO
  - newValue: string | null    // напр. новая дата в ISO
  - reason: string | null      // обязателен для date_changed
  - createdAt: timestamp

spaces/{spaceId}/tasks/{taskId}/attachments/{attachmentId}   // фото задачи и комментариев
  - attachedTo: "task" | "comment"
  - commentId: string | null   // заполнено, если attachedTo == "comment"
  - storagePath: string        // путь в Cloud Storage
  - downloadURL: string
  - contentType: string        // напр. "image/jpeg"
  - sizeBytes: number
  - width: number | null
  - height: number | null
  - uploadedBy: string         // uid
  - createdAt: timestamp
```

**Типы задач:**
- `scheduled` — исполнение привязано к конкретному `dueAt` (дата+время).
- `deadline` — отсчёт от `startAt` (= момент создания) до `dueAt`; UI показывает
  оставшееся время.

**Расположение файлов в Cloud Storage:**
```
spaces/{spaceId}/tasks/{taskId}/{attachmentId}.jpg
```

## 6. Бизнес-правила

1. **Перенос даты требует причины.** Изменение `dueAt` невозможно сохранить без
   непустого `reason`. Enforce в UI (кнопка «Сохранить» неактивна без текста) и
   дублировать проверку в Cloud Function/rules. Каждый перенос создаёт `event`
   типа `date_changed` с `oldValue`, `newValue`, `reason`.
2. **Любое изменение фиксируется в истории** (`events`): создание, смена статуса,
   перенос даты, комментарий, добавление вложения.
3. **Симметрия ролей.** Оба пользователя равноправны: любой может быть `createdBy`
   и любой — `assignedTo`. UI разделяет списки «Мне» (`assignedTo == me`) и
   «Я поставил(а)» (`createdBy == me`).
4. **Уведомление идёт партнёру, а не себе** — при генерации push исключать
   инициатора действия (`byUid`).
5. **Статусы:** `open` (по умолчанию), `completed` (выполнено), `declined` (не выполнено/отклонено).
6. **Приоритет.** Поле в интерфейсе называется «Степень важной важности», три значения:
   Низкий (`low`), Средний (`normal`, по умолчанию), Высокий (`high`). В списке отражается
   цветом/бейджем; доступна сортировка и фильтр по приоритету.
7. **Теги/категории** — общий для пары справочник (`spaces/{spaceId}/tags`), каждый с
   именем и цветом. На задаче хранится список `tagIds`. Доступен фильтр списка по тегам.
   Теги создаются/редактируются в настройках.
8. **Фотографии:** перед загрузкой изображение **сжимать** (макс. сторона ~1600 px,
   JPEG качество ~0.7, целевой размер файла до ~1–2 МБ). Показывать превью и
   индикатор загрузки. Одну задачу/комментарий не ограничивать одним фото — допускать
   несколько. При удалении задачи удалять и связанные файлы из Storage.

## 7. Экраны (SwiftUI)

1. **Auth / вход** — email + пароль. Регистрация создаёт `users/{uid}`.
2. **Привязка пары (pairing)** — один участник создаёт `space` и показывает `inviteCode`;
   второй вводит код → добавляется в `memberUids`, у обоих проставляется `spaceId`.
3. **Список задач** — сегменты «Мне» / «Я поставил(а)». Сортировка по `dueAt`/приоритету.
   Панель фильтров: по приоритету, по тегам, по статусу. Карточка: заголовок, дата/остаток
   времени, статус, цветной бейдж приоритета, чипы тегов, иконка вложения (если `hasAttachments`).
4. **Детали задачи** — заголовок, тип, даты, статус, приоритет, теги; **галерея фото**;
   действия: *отметить выполнено / не выполнено*, *перенести дату (с обязательной причиной)*,
   *добавить комментарий*, *прикрепить фото*; лента комментариев (с фото); история (`events`).
5. **Создание/редактирование задачи** — заголовок, заметки, выбор исполнителя (партнёр),
   тип (`scheduled` / `deadline`), `dueAt`, **«Степень важной важности»** (приоритет: Низкий/Средний/Высокий),
   **выбор тегов**, **прикрепление фото**.
6. **Комментарий** — поле текста + кнопка прикрепить фото (камера/галерея).
7. **Просмотр фото** — полноэкранный просмотрщик с зумом.
8. **Настройки** — профиль, статус привязки пары, **управление тегами** (создать/переименовать/
   цвет/удалить), разрешения на уведомления, выход.

## 8. Уведомления

**Push (через Cloud Functions → FCM), партнёру:**
- новая задача, назначенная на него → «Новая задача: …»;
- смена статуса задачи → автору;
- перенос даты → второй стороне, с текстом причины;
- новый комментарий → второй стороне.

**Локальные уведомления (на устройстве исполнителя):**
- напоминание в момент `dueAt` и, опционально, за N минут до него;
- при изменении/переносе задачи локальные напоминания перепланировать заново.

Реализация push-триггеров — Cloud Functions:
- `onTaskCreate` → уведомить `assignedTo`;
- `onTaskUpdate` → сравнить старое/новое: статус или дата изменились → уведомить партнёра;
- `onCommentCreate` → уведомить партнёра.

FCM-токен получать при запуске, сохранять в `users/{uid}.fcmTokens`, обновлять при ротации.

## 9. Правила безопасности

**Firestore:**
- Читать/писать документы внутри `spaces/{spaceId}/…` (включая `tasks`, `tags`,
  `comments`, `events`, `attachments`) может только пользователь, чей `uid` входит
  в `memberUids` этого `space`.
- `users/{uid}` доступен на запись только владельцу.
- Проверять корректность переходов статусов и наличие `reason` при `date_changed`.

**Cloud Storage:**
- Доступ к файлам по пути `spaces/{spaceId}/…` — только участникам этого `space`
  (проверка членства через Firestore/кастомные claims).
- Ограничить `contentType` изображениями и максимальный размер файла (напр. 5 МБ)
  прямо в Storage rules.

## 10. Пошаговый план реализации (для Claude Code)

Каждая фаза оставляет рабочий, проверяемый результат.

**Фаза 1 — Каркас проекта**
- Новый SwiftUI-проект в Xcode.
- Подключить Firebase SDK через SPM (`FirebaseAuth`, `FirebaseFirestore`, `FirebaseStorage`, `FirebaseMessaging`).
- Инициализировать Firebase, добавить `GoogleService-Info.plist`.

**Фаза 2 — Аутентификация и привязка пары**
- Экран входа/регистрации (email/пароль), создание `users/{uid}`.
- Создание `space` + `inviteCode`; ввод кода вторым участником; проставление `spaceId`.

**Фаза 3 — Слой данных**
- Модели (`Task`, `Comment`, `TaskEvent`, `UserProfile`, `Tag`, `Attachment`), репозитории.
- Realtime-листенеры на `tasks`, `comments`, `events`.

**Фаза 4 — Основной функционал задач (+ приоритеты, теги, фильтры)**
- Списки «Мне» / «Я поставил(а)», экран деталей, экран создания/редактирования.
- Оба типа задач, смена статуса, комментарии.
- Правило обязательной причины при переносе даты + запись `events`.
- Поле приоритета (бейджи, сортировка). Справочник тегов + управление в настройках.
- Панель фильтров (приоритет / теги / статус).

**Фаза 5 — Вложения (Cloud Storage)**
- Подключить Cloud Storage, настроить bucket.
- Выбор/съёмка фото (PhotosPicker + камера), сжатие перед загрузкой.
- Загрузка в Storage + запись метаданных в `attachments`, флаг `hasAttachments`.
- Галерея на экране задачи, фото в комментариях, полноэкранный просмотр.
- Удаление файлов из Storage при удалении задачи.

**Фаза 6 — Уведомления**
- Настройка FCM, регистрация и хранение токена.
- Локальные напоминания по `dueAt` (+ перепланирование при изменениях).
- Cloud Functions: триггеры на создание задачи / изменение / комментарий → FCM.

**Фаза 7 — Безопасность и полировка**
- Firestore и Storage security rules.
- Пустые состояния, обработка ошибок, часовые пояса, offline-режим.

**Фаза 8 — Установка на оба телефона**
- Подпись, provisioning; распространение через **TestFlight** (удобнее всего держать
  оба телефона на актуальной версии).

## 11. Предварительные ручные шаги (не код — выполнить до/во время Фаз 1, 5 и 6)

1. Оформить **Apple Developer Program** ($99/год). Нужен для push на реальном
   устройстве и для TestFlight. *(Mac есть, аккаунта пока нет — это первый шаг.)*
2. Установить Xcode.
3. Создать проект в **Firebase Console**: включить Firestore и Authentication (email/password),
   добавить iOS-приложение, скачать `GoogleService-Info.plist`.
4. Перевести проект Firebase на тариф **Blaze** (для Cloud Functions; по факту $0/мес).
5. Включить **Cloud Storage** в проекте Firebase (bucket по умолчанию).
6. В Apple Developer создать **APNs Auth Key (.p8)** и загрузить его в настройки
   Firebase Cloud Messaging — иначе push не заработает.
